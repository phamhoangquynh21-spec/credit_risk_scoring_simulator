# 💳 Credit Risk Scoring Simulator

Predict the probability that a credit-card customer **defaults on their next
payment** — and explain *why* — through an interactive dashboard. A Finance ×
Data Science portfolio project simulating a fintech / BNPL risk-assessment
workflow.

> **Status:** complete, runnable end-to-end, deploy-ready.
> **Author:** Adam Pham.

## About the data

This project uses a **synthetic dataset that mirrors the structure of the UCI
[*Default of Credit Card Clients*](https://archive.ics.uci.edu/dataset/350/default+of+credit+card+clients)
dataset** (Taiwan, 2005): identical schema (credit limit, sex, education,
marriage, age, 6 months of repayment status / bill / payment amounts), a
realistic ~24% default rate, and genuine, learnable relationships between
repayment behaviour and default. It is generated by `src/generate_data.py`.

**Why synthetic:** it keeps the repo free of any licensed/PII data while still
exercising the full pipeline. To use the real UCI file instead, drop its CSV in
`data/raw/credit_card_default.csv` (renamed to that path) — `clean_data()`
already handles the real category codes — and re-run training. See
[Limitations](#limitations).

## Quickstart

```bash
# 1. Create and activate a virtual environment (Python 3.11+)
python -m venv .venv
.venv\Scripts\activate            # Windows
# source .venv/bin/activate       # macOS/Linux

# 2. Install dependencies
pip install -r requirements.txt

# 3. Generate data + train models (writes models/model.pkl + metrics.json)
python -m src.train_model

# 4. (optional) Generate the reports (business report PDF + fairness audit)
python -m src.generate_reports

# 5. Launch the dashboard
streamlit run src/dashboard.py
```

The dashboard opens at `http://localhost:8501`.

## How it works

1. **Data** — `generate_data.py` creates 30k customers whose repayment delays and
   credit utilisation are driven by a latent "financial distress" factor; the
   default label is a logistic function of that factor plus noise, so features are
   genuinely predictive without being trivially separable.
2. **Preprocessing** — `clean_data()` fixes invalid category codes (EDUCATION
   0/5/6 → *other*, MARRIAGE 0 → *other*) and snake-cases columns;
   `engineer_features()` adds `avg_bill_amt`, `avg_pay_amt`, `credit_utilization`,
   `months_delayed_count`, `payment_trend`.
3. **Modeling** — Logistic Regression baseline (scaled, `class_weight='balanced'`)
   and an XGBoost/RandomForest advanced model tuned via 3-fold CV on AUC.
4. **Explainability** — `shap.TreeExplainer` yields per-customer top factors,
   surfaced in plain English.
5. **Fairness** — selection rate, recall and precision compared across sex,
   education and age groups; disparities disclosed (not silently mitigated).

## Limitations

- **Synthetic data** mirroring UCI Taiwan (2005) — *not real customers* and not
  calibrated to the Australian market.
- No transaction-level, credit-bureau, or macroeconomic signals.
- Class imbalance (~24%) — evaluated via **AUC-ROC**, not accuracy.
- A **false negative** (missed defaulter) is costlier than a false positive; the
  0.5 threshold should be tuned to real default-loss economics before any use.
- **This is a portfolio demonstration, not a production credit-decision system.**

Full fairness findings: [`reports/fairness_audit_report.md`](reports/fairness_audit_report.md).
Non-technical summary: [`reports/business_report.md`](reports/business_report.md).
